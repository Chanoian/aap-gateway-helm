## Project Overview

Production-grade Helm chart (`aap-gateway`) for deploying Ansible Automation Platform (AAP) on Red Hat OpenShift. The chart lives at the repo root (not in a `charts/` subfolder). Supported versions: `release-2.5`, `release-2.6`. Releases follow the same versioned branch model as the main AAP operator.

## Commands

```bash
# Lint the chart
helm lint .

# Lint strictly against all examples (what CI does — step 1)
for f in examples/*.yaml; do helm lint . --strict -f "$f"; done

# Render all examples (what CI does — step 2)
for f in examples/*.yaml; do helm template aap-gateway . -f "$f" > /dev/null; done

# Render a specific example
helm template aap-gateway . -f examples/full-stack.yaml

# Fetch fresh CRDs from the current oc context
./hack/fetch-crds.sh

# Regenerate values.schema.json from values.yaml + CRDs
python3 hack/gen-schema.py

# Delete all AAP CRDs from the cluster (use before switching operator versions)
./hack/delete-crds.sh

# Package the chart
helm package .

# Publish to Quay (triggered from GitHub Actions UI; or run locally for testing)
# Requires QUAY_USERNAME and QUAY_PASSWORD env vars set
helm registry login quay.io --username "$QUAY_USERNAME" --password-stdin <<< "$QUAY_PASSWORD"
helm package . --version <build-tag>
helm push aap-gateway-*.tgz oci://quay.io/achanoia  # helm appends chart name → pushes to quay.io/achanoia/aap-gateway

# Run tests for version extraction logic (no cluster required)
bash hack/test-version-extraction.sh
```

## Architecture

### Chart Structure
- `Chart.yaml` — `apiVersion: v2`, `name: aap-gateway`; `appVersion` is the AAP version string for the branch (e.g. `"2.5"`, `"2.6"`); `version` (chart semver) is independent
- `values.yaml` — single source of truth; every field has an inline comment; all optional features disabled by default; only `namespace` is required (empty string with `# REQUIRED` comment); `hostname` is optional — operator auto-generates a route hostname if omitted
- `values.schema.json` — JSON Schema (draft-07) generated by `hack/gen-schema.py`; derived from `values.yaml` + CRDs; enforces types and enums at `helm install`/`helm lint --strict` time
- `templates/_helpers.tpl` — `aap-gateway.labels`, `aap-gateway.resourceRequirements`, `aap-gateway.storageRequirements`, `aap-gateway.deepMerge`
- `crds/` — reference CRD definitions fetched from a live cluster via `hack/fetch-crds.sh`; used by `gen-schema.py` to derive the schema; NOT packaged with the chart (AAP Operator installs CRDs)
- `hack/fetch-crds.sh` — fetches AAP CRDs from the current `oc` context into `crds/`; also extracts the operator build version from the `operatorframework.io/installed-alongside-*` annotation and auto-patches `appVersion` in `Chart.yaml`; prints `Already up to date` and skips the patch when the version hasn't changed
- `hack/gen-schema.py` — generates `values.schema.json` by walking `values.yaml` and looking up types/enums in the CRDs; if `ansibleautomationplatforms.yaml` is absent (e.g. older AAP versions), falls back to type inference from defaults
- `hack/delete-crds.sh` — deletes all AAP CRDs from the cluster; use before installing a different operator version

### Key Template
`templates/aap.yaml` renders the single `AnsibleAutomationPlatform` CR (`apiVersion: aap.ansible.com/v1alpha1, kind: AnsibleAutomationPlatform`). The spec is built as a Go dict so `.Values.extraSpec` can deep-merge over any field cleanly. Each component section (controller, eda, hub) uses a direct pass-through loop: every key set under the component in `values.yaml` is emitted as-is into the component sub-spec, so any CRD field can be set without needing an explicit `extraSpec` block. The top-level `extraSpec` provides a final override layer for spec fields not tied to a specific component.

**Pass-through edge cases:**
- `kindIs "bool"` and `kindIs "float64"` guards are used throughout to preserve `false` and `0` values that would fail a simple truthiness check — always use these guards when adding new pass-through fields
- `hub.content` and `hub.worker` are handled separately from the top-level hub pass-through loop to avoid key duplication; their `replicas` fields are only emitted when not equal to the operator default of `2`
- Annotation fields (`route_annotations`, `ingress_annotations`, `service_annotations`, `service_account_annotations`) are serialized to JSON strings via `toJson` because the CRD declares them as `type: string`, not `type: object`

### values.yaml Conventions
- Use `{{- fail }}` for required field enforcement — missing values never silently produce broken manifests
- `extraSpec: {}` at top-level provides an escape hatch for spec fields not explicitly modeled; it is **deep-merged** (via `aap-gateway.deepMerge`), not shallow-overridden, so nested keys are merged recursively
- Empty string values are omitted from the rendered CR (only non-empty values make it into the manifest)

### Examples
`examples/` contains real-world patterns used directly in CI lint:
- `minimal.yaml` — only namespace; operator auto-generates hostname
- `controller-only.yaml` — hub and eda disabled
- `full-stack.yaml` — all components with internal DB/Redis
- `external-db.yaml` — external database secret pattern
- `full-stack-external-db-resources.yaml` — full stack with resource limits
- `hub-advanced.yaml` — exercises hub nested fields (`content.*`, `worker.*`)
- `explicit-zero-replicas.yaml` — verifies `replicas: 0` passes through correctly
- `complex-crd-coverage.yaml` — exhaustive CRD field coverage across all components
- `testing-full-stack.yaml` — all components, labels/annotations on all resources, resource limits, internal DB, real-deployment validated

### Release Strategy
- `main` is a read-only pointer to the latest release branch — **never commit directly to `main`**
- Each AAP minor version gets a `release-2.x` branch (`release-2.5`, `release-2.6`, etc.)
- All work (fixes, schema updates, CRD updates) goes to the appropriate release branch first
- `main` is fast-forward merged from the latest release branch after changes land there:
  ```bash
  git checkout main
  git merge --ff-only release-2.6
  git push origin main
  ```
- Bug fixes that apply to older branches are cherry-picked: `git cherry-pick <sha>`
- CI runs `helm lint --strict` and `helm template` on every PR against `main` and `release-*`

### Per-Build Release Workflow
When AAP ships a new build:
1. `git checkout release-2.6`
2. `./hack/fetch-crds.sh` — fetches fresh CRDs **and** auto-updates `appVersion` in `Chart.yaml` from the operator annotation; prints `Already up to date: <version>` if nothing changed
3. `python3 hack/gen-schema.py` — regenerate `values.schema.json`
4. Update `values.yaml` if the CR spec added new fields worth exposing
5. `for f in examples/*.yaml; do helm lint . --strict -f "$f"; done && for f in examples/*.yaml; do helm template aap-gateway . -f "$f" > /dev/null; done` — must pass
6. Commit, tag (`chart-2.6.0-0.1774648945`), push
7. Fast-forward `main`: `git checkout main && git merge --ff-only release-2.6 && git push origin main`
8. Trigger the **Helm Publish** workflow from the GitHub Actions UI

### Constraints
- No subcharts or Helm dependencies
- No Kustomize
- No hardcoded environment-specific values
- Minimum supported AAP version: 2.5 (`release-2.5` branch)

### Known Operator Bugs (as of AAP 2.5/2.6)
- **`additional_labels` / `service_labels` not propagated:** The `additional_labels` field on `AutomationController` and `EDA` CRDs is accepted at the CR level but the operator does not apply these labels to pods, deployments, or services. `service_labels` on `AutomationController` has the same issue. Hub has no label fields at all. Fields are kept in `examples/testing-full-stack.yaml` to validate once the operator is fixed.
- **String-typed complex fields in child CRDs:** `node_selector`, `topology_spread_constraints`, and all `*_annotations` fields on `AutomationController`, `EDA`, and `AutomationHub` are declared as `type: string` (JSON-encoded) in their CRDs. Passing these as native YAML objects/arrays via the component pass-through causes the child CR admission webhook to reject them. Always set these via `extraSpec` as pre-serialized JSON strings.
