## Project Overview

Production-grade Helm chart (`aap-gateway`) for deploying Ansible Automation Platform (AAP) on Red Hat OpenShift. The chart lives at the repo root (not in a `charts/` subfolder). Compatible with AAP 2.5 and 2.6+. Chart versioning is independent of the AAP operator version ‚Äî a single chart works across operator versions.

## Commands

```bash
# Lint the chart
helm lint .

# Lint strictly against all examples (what CI does ‚Äî step 1)
for f in examples/*.yaml; do helm lint . --strict -f "$f"; done

# Render all examples (what CI does ‚Äî step 2)
for f in examples/*.yaml; do helm template aap-gateway . -f "$f" > /dev/null; done

# Render a specific example
helm template aap-gateway . -f examples/full-stack.yaml

# Fetch fresh CRDs from the current oc context (specify AAP version)
./hack/fetch-crds.sh 2.6   # saves to crds/2.6/

# Regenerate values.schema.json from values.yaml + CRDs (defaults to latest version)
python3 hack/gen-schema.py
python3 hack/gen-schema.py --version 2.5  # use specific version

# Delete all AAP CRDs from the cluster (use before switching operator versions)
./hack/delete-crds.sh

# Package the chart
helm package .

# Publish to Quay (triggered from GitHub Actions UI; or run locally for testing)
# Requires QUAY_USERNAME and QUAY_PASSWORD env vars set
helm registry login quay.io --username "$QUAY_USERNAME" --password-stdin <<< "$QUAY_PASSWORD"
helm package . --version <build-tag>
helm push aap-gateway-*.tgz oci://quay.io/achanoia  # helm appends chart name ‚Üí pushes to quay.io/achanoia/aap-gateway

# Run tests for version extraction logic (no cluster required)
bash hack/test-version-extraction.sh
```

## Architecture

### Chart Structure
- `Chart.yaml` ‚Äî `apiVersion: v2`, `name: aap-gateway`; `appVersion: ">=2.5"` (informational, chart supports multiple operator versions); `version` follows independent chart semver
- `values.yaml` ‚Äî single source of truth; every field has an inline comment; all optional features disabled by default; only `namespace` is required (empty string with `# REQUIRED` comment); `hostname` is optional ‚Äî operator auto-generates a route hostname if omitted
- `values.schema.json` ‚Äî JSON Schema (draft-07) generated by `hack/gen-schema.py`; derived from `values.yaml` + CRDs; enforces types and enums at `helm install`/`helm lint --strict` time
- `templates/_helpers.tpl` ‚Äî `aap-gateway.labels`, `aap-gateway.resourceRequirements`, `aap-gateway.storageRequirements`, `aap-gateway.deepMerge`
- `crds/<version>/` ‚Äî reference CRD definitions per AAP version (e.g. `crds/2.5/`, `crds/2.6/`); fetched from a live cluster via `hack/fetch-crds.sh <version>`; used by `gen-schema.py` to derive the schema; NOT packaged with the chart (AAP Operator installs CRDs)
- `hack/fetch-crds.sh <version>` ‚Äî fetches AAP CRDs from the current `oc` context into `crds/<version>/`; also reports the detected operator build version from the `operatorframework.io/installed-alongside-*` annotation
- `hack/gen-schema.py` ‚Äî generates `values.schema.json` by walking `values.yaml` and looking up types/enums in the CRDs; accepts `--version 2.5` or `--version 2.6` (defaults to latest version directory); if `ansibleautomationplatforms.yaml` is absent, falls back to type inference from defaults
- `hack/delete-crds.sh` ‚Äî deletes all AAP CRDs from the cluster; use before installing a different operator version

### Key Template
`templates/aap.yaml` renders the single `AnsibleAutomationPlatform` CR (`apiVersion: aap.ansible.com/v1alpha1, kind: AnsibleAutomationPlatform`). The spec is built as a Go dict so `.Values.extraSpec` can deep-merge over any field cleanly. Each component section (controller, eda, hub) uses a direct pass-through loop: every key set under the component in `values.yaml` is emitted as-is into the component sub-spec, so any CRD field can be set without needing an explicit `extraSpec` block. The top-level `extraSpec` provides a final override layer for spec fields not tied to a specific component.

**Pass-through edge cases:**
- `kindIs "bool"` and `kindIs "float64"` guards are used throughout to preserve `false` and `0` values that would fail a simple truthiness check ‚Äî always use these guards when adding new pass-through fields
- `hub.content` and `hub.worker` are handled separately from the top-level hub pass-through loop to avoid key duplication; their `replicas` fields are only emitted when not equal to the operator default of `2`
- Annotation fields (`route_annotations`, `ingress_annotations`, `service_annotations`, `service_account_annotations`) are serialized to JSON strings via `toJson` because the CRD declares them as `type: string`, not `type: object`

### values.yaml Conventions
- Use `{{- fail }}` for required field enforcement ‚Äî missing values never silently produce broken manifests
- `extraSpec: {}` at top-level provides an escape hatch for spec fields not explicitly modeled; it is **deep-merged** (via `aap-gateway.deepMerge`), not shallow-overridden, so nested keys are merged recursively
- Empty string values are omitted from the rendered CR (only non-empty values make it into the manifest)

### Examples
`examples/` contains real-world patterns used directly in CI lint:
- `minimal.yaml` ‚Äî only namespace; operator auto-generates hostname
- `controller-only.yaml` ‚Äî hub and eda disabled
- `full-stack.yaml` ‚Äî all components with internal DB/Redis
- `external-db.yaml` ‚Äî external database secret pattern
- `full-stack-external-db-resources.yaml` ‚Äî full stack with resource limits
- `hub-advanced.yaml` ‚Äî exercises hub nested fields (`content.*`, `worker.*`)
- `explicit-zero-replicas.yaml` ‚Äî verifies `replicas: 0` passes through correctly
- `complex-crd-coverage.yaml` ‚Äî exhaustive CRD field coverage across all components
- `testing-full-stack.yaml` ‚Äî all components, labels/annotations on all resources, resource limits, internal DB, real-deployment validated
- `vso.yaml` ‚Äî VSO (Vault Secrets Operator) integration; AppRole auth; five secrets opted in (admin password, gateway DB, controller DB, EDA DB, Redis)

### Release Strategy
- All work happens on `main` ‚Äî one branch, no version-specific branches
- Chart version (`version` in `Chart.yaml`) is independent of AAP operator version ‚Äî bump it for each release
- CI runs `helm lint --strict` and `helm template` on every PR against `main`

### Per-Build Release Workflow
When AAP ships a new build (e.g. 2.6):
1. `git checkout main`
2. `./hack/fetch-crds.sh 2.6` ‚Äî fetches fresh CRDs into `crds/2.6/` and reports the detected operator build version
3. `python3 hack/gen-schema.py` ‚Äî regenerate `values.schema.json` (uses latest CRD version by default)
4. Update `values.yaml` if the CR spec added new fields worth exposing
5. `for f in examples/*.yaml; do helm lint . --strict -f "$f"; done && for f in examples/*.yaml; do helm template aap-gateway . -f "$f" > /dev/null; done` ‚Äî must pass
6. Bump `version` in `Chart.yaml` (e.g. `1.0.0` ‚Üí `1.1.0`)
7. Commit, push, trigger the **Helm Publish** workflow from the GitHub Actions UI

### Constraints
- No subcharts or Helm dependencies
- No Kustomize
- No hardcoded environment-specific values
- Minimum supported AAP version: 2.5

### Known Operator Bugs (as of AAP 2.5/2.6)
- **`additional_labels` not propagated:** The `additional_labels` field on `AutomationController` and `EDA` CRDs is accepted at the CR level but the operator does not apply these labels to pods, deployments, or services. Hub has no label fields at all. Fields are kept in `examples/testing-full-stack.yaml` to validate once the operator is fixed.
- **`service_labels` on `AutomationController` causes reconciliation failure:** The operator embeds the JSON string value of `service_labels` as a raw YAML scalar when generating the Service manifest. Because the value contains colons (e.g. `{"app.kubernetes.io/part-of":"aap"}`), the YAML parser treats it as a mapping key and fails with `could not find expected ':'`. This breaks `Apply Resources` and prevents controller pods from being created. **Do not set `service_labels` on the controller** ‚Äî it is commented out in `examples/testing-full-stack.yaml`.
- **String-typed complex fields in child CRDs:** `node_selector`, `topology_spread_constraints`, and all `*_annotations` fields on `AutomationController`, `EDA`, and `AutomationHub` are declared as `type: string` (JSON-encoded) in their CRDs. Passing these as native YAML objects/arrays via the component pass-through causes the child CR admission webhook to reject them. Always set these via `extraSpec` as pre-serialized JSON strings.
